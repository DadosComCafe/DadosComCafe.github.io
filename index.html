  • Dados com Café

    Em uma das minhas oportunidades como Engenheiro de Dados, percebi que algumas das equipes não técnicas precisavam de mais acesso aos dados, uma vez que alguns relatórios eram elaborados por eles próprio.
    Então, percebendo o anceio destes, resolvi então dispor de 1 hora semanalmente para dar aulas de SQL, assim como docker para reproduzir o ambiente de produção na máquina local.
    Desta forma, ao invés de me pedirem para elaborar uma query para acessar tais dados, me trouxessem a query pronta,e eu apenas a executava no banco de produção.
    Daí surgiu o canal no youtube: Dados Com Café, onde eu publico vídeos técnicos de como modelar bancos de dados, consultar tabelas, reproduzir o ambiente na máquina local, não apenas SQL, mas todo o envolto de banco de dados.
    O canal pode ser acessado aqui: Dados Com Café


  • Consumindo api do Starwar, e modelando com pydantic

    O Pydantic é uma ferramenta que é usada para a validação de dados. Como no python, por padrão não se declara o tipo das variávies, o pydantic pode resolver esta questão.
    Então, obtendo os valores da api do starwars, através da biblioteca requests, eu os recebo, e os aplico a models. Desta forma, cria-se um objeto pydantic para cada valor que é recebido da api. Então, declara-se a forma dos dados como classes com atributos. A vantagem, do objeto pydantic é a garantia de que ele vai corresponder a tipagem esperada, do contrário, vai quebrar. Quanto antes quebrar um dado errado, melhor! Outra coisa bastante interessante que é apresentada neste repositório, é a facilidade de implementar herança entre os objetos pydantic, o que faz total sentido dependendo do tipo de dados.
    O repositório pode ser acessado aqui: swapi_study


  • ETL marota com Airflow, GCP

    Neste projeto, desenvolvo um ETL, e crio bancos de dados com o docker: -se inicia com uma task que faz download do dataset, onde o nome do dataset o do author são consultados no mongo, então os arquivos do zip baixado são enviados para o Storage, para ser usado como DataLake. Os dados do csv são "tratados", e estes dados, agora com tipos específicos, já que inicialmente são todos textos, são então enviados para um banco de dados postgres local. Estes dados do postgres, são então migrados para o BigQuery, onde fazendo integração com o Storage, é utilizado como DataWarehouse. O projeto ainda não terminou, portanto outras ferramentas do GCP também podem começar a ser utilizadas. O repositório pode ser acessado aqui:Airflow ETL
